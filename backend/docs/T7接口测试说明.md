## 一、先看这一条，能省掉一半误报

**除了越权，所有失败都返回 HTTP 200，业务结果看 body 里的 `code` 字段。**

```json
{ "code": 3101, "message": "存档已满（最多 5 个），请先删除已有存档", "details": null }
```

这是有意为之：微信小程序的 `wx.request` 遇到非 2xx 响应不走 `fail` 回调，用「HTTP 200 + 业务码」前端更好处理。

**唯一的例外**是越权访问他人资源，返回真实的 **HTTP 403 + `code: 1004`**。这一条是按你们文档 FN-11-08 / BR-26 / RSK-03 的要求实现的。

所以：看到 HTTP 200 不等于用例通过，要看 `code`。看到 `code` 不为 0 也不等于有 bug，很多是预期的业务拒绝。

> 补充：认证失败（`1002` 缺 token、`1003` token 无效或过期）目前也返回 HTTP 200。要不要一并改成 401，是待澄清事项 Q-21，还等架构拍板。若最终定为 401，改的是后端 A 的中间件，届时验收口径要同步调整。

## 二、错误码

| code | HTTP | 含义 |
| --- | --- | --- |
| `0` | 200 | 成功 |
| `1002` | 200 | 缺少认证 token |
| `1003` | 200 | token 无效或已过期 |
| `1004` | **403** | 无权操作该资源（越权） |
| `2001` | 200 | 缺少必要参数 |
| `2002` | 200 | 参数非法 |
| `3001` | 200 | 管理员用户名或密码错误 |
| `3002` | 200 | 管理员账号已锁定（15 分钟） |
| `3101` | 200 | 存档已满 |
| `3102` | 200 | 短剧情不支持存档 |
| `4101` | 200 | 剧情不存在或暂未开放 |
| `4102` | 200 | 剧情节点不存在 |
| `4103` | 200 | 存档不存在 |
| `4104` | 200 | 地图资源未配置 |
| `4105` | 200 | 暂无可导出数据（**不生成空文件**） |
| `5000` | 200 | 服务器内部错误 |

分段规则：`1000` 认证与权限，`2000` 参数，`3000` 业务规则，`4000` 资源不存在，`5000+` 服务端异常。
其中 `3001-3099` / `4001-4099` 归后端 A，`3100+` / `4100+` 归后端 B。

## 四、账号

| 角色 | 凭证 |
| --- | --- |
| 普通用户 | `POST /api/auth/wechat-login`，body `{"code":"任意字符串"}` |
| 管理员 | `admin` / `rjks@bjut514` |

**当前是模拟登录**，`code` 传什么都能登录，同一个 `code` 对应同一个用户。真实的微信 `code2Session` 由后端 A 后续接入，那之前测不了 FN-01 的真实授权链路。

管理员连续错 5 次会锁 15 分钟（PRD F-10）。测完锁定用例后，执行下面这条 SQL 立即解锁，不用等 15 分钟：

```sql
UPDATE admins SET failed_count = 0, locked_until = NULL WHERE username = 'admin';
```

`test/t7-selftest.js` 每次运行前会自动做这件事，所以跑脚本不受影响。

## 六、以下行为是设计如此，请勿记为缺陷

1. **`DELETE /api/save/{id}` 不存在。** 你们文档 FN-11-03 写的是这个路径，接口规范（《软件设计说明》4.2）是 `DELETE /api/save-slots/{slotId}`。以后者为准。

2. **没有「存档详情」接口。** FN-11-02 假定存在 `GET /api/save-slots/{id}`，接口规范里没有。`GET /api/save-slots` 的列表响应已经带了 `nodeId`，前端直接用即可，不需要单独的详情接口。

3. **已完成的办事流程可以重新进入。** FN-07-08 写的是「前端校验流程状态为未完成才允许进入」，但待澄清事项 Q-01 的结论是「支持已完成的流程重新进入体验，前置条件仅为用户已登录、位于办事流程选择页」。后端不以完成状态阻止进入，重进也不重复发勋章。FN-07-08 需要同步修订。

4. **导出没有做 >1000 条的分批与进度条。** FN-15-06 要求这个，但项目成功标准是 10 人注册（你们自己的 RSK-13 也提了）。V1.0 单批导出，建议此项降级。

5. **存档列表没有分页。** FN-11-11 要求列表过长时分页，但 BR-23 规定单用户最多 5 个存档，「过长」不可能发生。

6. **`card` / `print` 两个办事流程不返回。** Q-09 已定为 V1.0 隐藏。剧情数据在库里（后台预览和导出能看到），但用户端接口不下发。
